# Threads content and scheduling

Use this skill when the user wants to build a Threads presence, draft Threads posts, turn a positioning idea into a thread, or schedule Threads content through an API/scheduler.

## Content style for this user

- Write in Indonesian unless the user asks otherwise.
- Prefer natural, human language over technical labels. If the audience is non-technical, avoid terms like `scope`, `product engineering`, `project engineering`, `MVP`, or `technical discovery` unless the user explicitly wants them.
- Translate technical value into everyday client language:
  - `scope` → "bagian yang paling penting untuk dibuat"
  - `requirements` → "kebutuhan"
  - `product engineering` → "membantu merapikan ide lalu membangun aplikasinya"
  - `project engineering` → "membantu project digital berjalan lebih jelas dan bisa dieksekusi"
- For service positioning posts, lead with the reader's felt problem before selling the service. Strong angle from this session: "ide yang mati pelan-pelan karena terlalu lama dipikirkan sendirian."
- When Basim asks for posts that "memancing emosional", write for recognition first: make the reader feel "ini gue banget" through quiet pains, delayed ideas, hidden fatigue, guilt, fear of being judged, or regret after seeing someone else build a similar idea. Keep it humane, not manipulative.
- Basim's Threads/Repliz post bodies should not include hashtags by default. If a platform supports topic tags, keep them out of the visible body unless Basim explicitly asks for hashtags/tags.
- Avoid sounding like a brochure. Use short paragraphs, plain words, and a gentle call to action: "mungkin sekarang waktunya ngobrol" / "kita bisa mulai dari ngobrol dulu."
- When Basim asks to turn event notes/PDF recap into a “draft catatan acara”, do **not** over-convert it into a viral storytelling essay by default. Preserve the event-note purpose: clear recap, useful bullets/short thread parts, professional but natural tone, and tag people who asked for the recap (e.g. `@username`) when requested. If he says someone asked to share the results, write it as a shareable event note/reply first; offer a more viral version separately only if useful.
- When the user asks for spontaneous/viral Threads drafts, make them feel like a casual human chat or curhat, not a polished template. Use slightly imperfect hooks such as “barusan kepikiran”, “ngeliat rame soal…”, “kayaknya…”, or “jujur…”. Keep the logic clear but avoid over-structured headings inside the post body.
- For very short variants, honor constraints literally (e.g. “buat 2 kalimat” means produce two-sentence options, not a long framework). Keep the punch in the first sentence and the relatable conflict or question in the second.
- When connecting to viral/current topics, search first and cite/use concrete trend context lightly (e.g. what is viral, platform/tag scale if available). Do not force random news; tie it back to digital behavior, tools, business systems, privacy, owner fatigue, app/project execution, or beginner AI usage.
- Basim correction for scheduled Threads workflows: before posting or drafting growth content, search current Google/web + Indonesian news/trend signals first, especially politics/government, demos/protests, public policy, citizen-impact stories, economic pressure, security/digital incidents, and viral lived-experience stories. Use these as story seeds, not raw news summaries. Adapt @nafkahnation.dx mechanics: human case -> victim/loss -> twist -> hidden system -> repostable lesson. Do not reduce posting cadence just because the audit says quality must improve.
- For Basim current-news/newsjacking posts, raw facts alone are usually too flat. After verifying the numbers, translate them into everyday emotional consequences: “baru terasa pas berdiri di SPBU”, “rasa mikir sebelum keluar rumah”, “uang makan/pulsa/parkir kegeser”, or “budget keluarga yang sudah presisi.” Keep the emotion grounded in concrete costs/choices, not melodrama.
- For disaster/public-safety news (earthquakes, tsunami warnings, floods, etc.), prioritize usefulness and sensitivity over clever positioning. Verify against official or near-official reporting where possible (BMKG/BPBD/BNPB/government; reputable media quoting official briefings), include the concrete alert data early (magnitude, time, affected warning statuses/areas, official recommendation), and add a caveat that warning status can change quickly. Avoid opportunistic sales angles, jokes, or engagement-bait. Best-performing structure for Basim: short emotional hook → data peringatan → practical tutorial/checklist → official-source reminder. Tone: humane, calm, non-alarmist, and shareable.
- For beginner AI tips/tricks content, use simple everyday examples rather than prompt-engineering jargon. Anchor on the practical lesson: clear context beats clever prompts; include “buat siapa, tujuannya apa, gaya bahasanya gimana, formatnya mau apa.” See `references/beginner-ai-content-notes.md`.
- When Basim asks for `tema prioritas`, brainstorming, or viral themes per social platform, do not draft immediately from generic trends. First combine recent Repliz performance, light current trend research, Basim positioning fit, and platform-native format. Use `references/basim-platform-theme-prioritization.md` for the priority clusters and platform mapping.

## Threads copy workflow

Threads creator best practices synthesized from Meta's official Threads for Creators guidance and Basim account audits:
- Treat Threads as a conversation platform, not a broadcast channel. Replies help people in the conversation discover the profile; replying to replies and weighing in on other relevant threads are part of growth, not optional aftercare.
- The For You feed is influenced by likes, replies, reposts, prior interactions across Threads/Instagram, hidden/muted/reported activity, and profiles clicked/followed. Optimize for real conversation and profile discovery, not just likes.
- Distribution can be limited by non-recommendable, unoriginal, clickbait, engagement-bait, contest/giveaway-style, or overly promotional content. Prefer original lived-experience stories, useful commentary, and specific opinions.
- Threads supports text posts up to 500 characters, replies, reposts, quote posts, media/video up to 5 minutes, carousels/albums up to 20 images/videos, polls, and one topic tag per post. Always include text with media/carousel posts.
- Growth levers: post often, reply often, own a niche, use one relevant topic tag, cross-promote to Instagram Stories/feed when useful, jump on current events only when there is a genuine angle, and use Insights/View activity to compare views, replies, reposts/quotes, and follower/non-follower reach.
- For Basim, prioritize current Indonesian news/trend research before drafting, then turn it into human-case storytelling rather than raw news. Use data only as context after the story hook.
- Basim correction: do **not** add hashtags to Threads/Repliz post bodies. Avoid visible `#...` tags and do not append generic hashtag blocks at the end. If Repliz/Threads topic metadata is needed, keep it out of the visible body unless Basim explicitly asks.

For emotional Threads patterns and proven hooks, see `references/emotional-threads-patterns.md`.
- For Basim's viral radar / growth-first Threads workflow — search first, score topics, separate evergreen auto-schedule from current-event approval queue — see `references/basim-viral-radar-growth-workflow.md`.
For Basim's aggressive but compliant growth mode — high-volume evergreen Threads posts, approval-only current events, manual engagement briefs, daily throttle rules, and 3-day trial/audit mode — see `references/basim-aggressive-human-like-publishing.md`. When Basim asks for “brutal posting” or growth without looking like a bot, do **not** provide evasion/bypass tactics; implement aggressive human-like publishing: official scheduling, natural hook rotation, no mass-follow/like/comment automation, no bursts, and safety gates. If Basim says a support/brief job “ga usah ditampilin lagi”, pause or remove that output job rather than continuing to deliver operational briefs; keep core posting/radar jobs unchanged unless he names them.
For Basim's adaptation of narrative/viral storytelling patterns learned from @nafkahnation.dx, see `references/nafkahnation-storytelling-patterns.md`: start with a concrete person, expose the victim/loss, add a twist/reversal, then explain the hidden system/mechanism. When Basim asks to make storytelling “untuk yang dijadwalkan di Repliz”, treat the scheduled post’s hook/angle as the seed for a 10-part thread; do not assume it should replace the existing scheduled item unless he explicitly asks to reschedule/replace it.
For Basim carousel cron quality and prompt-regression recovery, see `references/basim-carousel-cron-quality-audit.md`: keep the 15 June-style audit/scoring/report, do not let provider/model migration collapse the prompt into a generic short run, and preserve no-hashtag body rules.

1. Identify the target reader and pain:
   - founder with an app idea
   - business owner needing an internal system
   - team with a stuck digital project
2. Start from a relatable moment, not a credential claim.
3. Keep each post under Threads' character limit. For multi-post threads, keep each post self-contained but connected.
4. If Repliz scheduling is likely, keep each main post conservatively under **500 characters** unless using thread replies. Actually count characters with a tool/script before presenting “final” drafts or sending them for review; do not rely on eyeballing.
5. If the user asks to audit drafts before sending to an update/review topic, provide:
   - per-post status (strong/medium/weak), risk, and edit recommendation,
   - character counts,
   - a final revised set with every post under the chosen limit,
   - priority order for posting.
6. For this user’s Threads performance, prefer a mix of short/spontaneous posts and practical prompts over ten polished essays. Recent pattern: overly polished reflective posts produced low engagement, while shorter contextual/spontaneous posts performed better. Add “pintu komentar” CTAs such as “Yang paling kerasa di kalian?”, “Ada yang relate?”, or “Bagian mana yang masih manual?”
7. When learning from high-performing creator/storytelling examples, adapt the underlying content mechanics rather than copying the voice: open with a concrete human situation, show who is harmed or losing time/money/energy, reveal the hidden system/mechanism, and use a reversal such as “dikira masalahnya A, ternyata B.” For Basim, map this to AI setup friction, owner-as-manual-system fatigue, Meta Ads follow-up leaks, and automation bottlenecks.
- When turning viral/current topics into drafts, map each topic back to Basim’s positioning (digital, systems, business, execution, app ideas, owner fatigue). Avoid random news angles that do not support the account direction.
- When Basim explicitly says to set aside his own profile/positioning for faster Threads growth, switch from positioning-fit first to a **viral radar / bongkar-sistem workflow**: search current trend signals first, score candidate topics for momentum/relatability/shareability/twist/safety/fit, then draft only the strongest 3 candidates. See `references/basim-viral-radar-growth-workflow.md`.
- For newsjacking/current-event growth posts, do not auto-schedule from the first request. Output exact wording + audit + risk status and ask Basim to approve a numbered candidate before any Repliz scheduling.
8. If the user provides posts labeled `Post 1`, `Post 2`, etc., preserve their order and schedule as a nested/thread reply structure when the API supports it.
9. When the user says not to post an older draft, treat that as a hard constraint. Do not reuse earlier drafts unless they explicitly approve.

## Repliz scheduling workflow

Repliz can schedule text posts, nested Threads, and carousel/album posts via `POST https://api.repliz.com/public/schedule` using Basic Auth. See `references/repliz-schedule-api.md` for the request shapes captured from the docs, including `type: "album"` for multi-image carousel posts where each media item must be a public URL.

For Repliz account discovery and Basim multi-platform carousel defaults (Threads/Instagram album, TikTok/YouTube MP4 slideshow, verified account IDs, and cron pause-before-debugging safety), see `references/repliz-account-discovery-and-multiplatform.md`.

Hard approval gate for Basim/Repliz:
- Before any `POST /public/schedule` call, audit the final drafts first: fit to positioning, risk of sounding templated/brochure-like, character count, CTA/pintu komentar, and whether topic/title metadata might leak into the Threads body.
- Do **not** send/schedule anything to Repliz until Basim explicitly approves the audited final draft in the chat (e.g. “approve”, “oke kirim”, “schedule”, “post jadwalkan ke repliz”).
- Scheduled/cron jobs must output analysis + audited drafts only; they must not auto-schedule to Repliz unless their prompt explicitly says approval has already been given.
- When approved, schedule only the post body. Do not include internal topic/title labels as Threads content.
- Destination pitfall: in a Basim/Repliz context, “kirim”, “post”, or “jadwalkan” usually means send/schedule to Repliz once approval is clear. Do **not** send to the internal Telegram “update social media” topic unless the user explicitly asks for review/update topic delivery.
- If the user gives a batch of numbered posts and asks to add “utas kedua” to each, create separate scheduled Threads where each main post has the same CTA as `replies[0].description`.
- If Basim asks to schedule immediately after drafting (e.g. “buatkan 10 draft ... dan kirimkan schedule ke Repliz”), this may count as approval only for low-risk evergreen/batch content where he is clearly asking to schedule a freshly generated artifact. Still audit/count characters before scheduling, schedule only post bodies, and report Repliz schedule IDs.
- **Basim news/current-event exception:** for “cari berita sekarang ... lalu post” or similar newsjacking/current-affairs requests, do **not** treat the initial “post” as final approval. First research, draft, audit, and send the exact wording back in chat. Schedule to Repliz only after a second explicit approval on that wording (e.g. “oke post”, “gas post”, “kirim ke Repliz”). If you accidentally scheduled too early and Basim asks to review first, immediately cancel the pending schedule, then provide a revised draft.
- If the user asks to schedule but gives no exact times, use a sensible default cadence instead of asking when low-stakes: next near-future 15-minute slot (at least ~10 minutes ahead) and then every 90 minutes, reporting all local WIB times and schedule IDs.
- If Basim asks to revise already-scheduled Repliz posts, fetch the existing schedules, preserve their `scheduleAt`, and update safely. If direct `PUT/PATCH` is unsupported, delete the old schedule and recreate the revised one at the same time; verify via `GET /schedule/{newScheduleId}` and report new IDs.
- For Basim storytelling/nested Threads, avoid visible numeric prefixes (`1/`, `2/`, etc.) at the beginning of each part unless explicitly requested. The Repliz reply structure already preserves order; strip numbering when he asks to remove “angka didepannya”.

Before calling the API, collect:
- Repliz access key
- Repliz secret key
- Threads `accountId`
- final post text
- target time and timezone

Security:
- Do not save API secrets in memory or skills.
- If secrets were pasted into a chat, advise the user to rotate/regenerate the secret after the task.
- Use tool output only to report status code and schedule ID. Do not echo secrets back.

Scheduling time:
- For WIB, convert to UTC by subtracting 7 hours. Example: 11:10 WIB → 04:10 UTC.
- If the requested time today has already passed, use the next occurrence unless the user says a date.

Verification:
- A successful Repliz create schedule call may return HTTP `201` with a JSON body containing `scheduleId`, even if the docs show `200`.
- Report the scheduled local time, the platform, and `scheduleId`.
